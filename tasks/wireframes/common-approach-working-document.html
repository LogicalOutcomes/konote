KoNote & CIDS Full Tier: Where We Are

Working document for Common Approach colleagues · March 2026
Prepared by Gillian Kerr, Logical Outcomes

KoNote is a participant outcome management system designed for Canadian nonprofits — each agency runs their own instance. We're building toward CIDS Full Tier compliance and wanted to share where we are and what we've learned mapping the standard so far.

Where we are right now

The software supports all 14 CIDS Full Tier classes, but we haven't yet tested the full pipeline with real program data. Here's what's built and what still needs work:

Built

  • Programs carry CIDS sector codes and population served codes
  • Metrics and outcome targets carry CIDS indicator fields
  • Code list support for ICNPO, PopulationServed, SDG, and IRIS+ (based on 5.3b — not yet populated on our test instance)
  • Taxonomy classification workflow — AI suggests mappings to code lists; an admin reviews and approves them
  • Secure export with approval flow — exports are reviewed before they leave the instance, and any personally identifiable information is flagged
  • Aggregate reports suppress small groups (k≥5) to protect privacy
  • Agency profile for organisation-level CIDS metadata
  • Basic Tier JSON-LD export with SHACL validation
  • Full Tier JSON-LD export — assembles all 14 CIDS classes into one document
  • IndicatorReport computes outcome statistics per metric type: success rates, means, medians, distributions, pre/post change, effect sizes
  • Data quality signals on IndicatorReport using W3C DQV — reporting rates, observation density, plus structured descriptors for evidence type, measure basis, and derivation method (describe, don't rank)
  • Evaluation framework editor — agencies define their program's services, activities, outputs, risks, and counterfactuals, which map directly to Full Tier classes
  • Enrichment — derives themes and standards alignment from program data
  • Evaluator attestation — optional step where an evaluator reviews the framework before export
  • Coverage dashboard — shows which programs have class coverage and where gaps remain

Still to do

  • Test the full pipeline with real program data
  • Full Tier SHACL validation (we'd like to use your improved validation tool when it's available)

Privacy

KoNote is designed for Canadian nonprofits that handle data subject to PHIPA (Ontario) and comparable provincial health privacy legislation. All personally identifiable information is encrypted locally and does not leave the instance.

Full Tier CIDS classes describe program models and aggregate outcomes, not individual participants. cids:Stakeholder refers to stakeholder groups (e.g., "youth aged 16-24"), not individuals. cids:StakeholderOutcome describes aggregate outcomes for a group. cids:ImpactModel is the theory of change.

Our exports work in three layers:

LayerWhat's in itPrivacy
1. Program model ImpactModel, Service, Activity, Output, Stakeholder groups, ImpactRisk, Counterfactual No concern — describes programs, not people
2. Aggregate measurement IndicatorReport with cohort results, suppressed to k≥5 Statistical disclosure controls
3. Case trajectories (optional) De-identified outcome pathways per cohort k-anonymity, dates generalised to quarters, minimum n≥15

But can you trust the numbers?

Completing all 14 CIDS classes tells a funder that a report is structurally complete. It says nothing about whether the underlying data is reliable. A program could populate every class while having a 20% response rate, unvalidated measurement tools, and self-selected respondents. A funder aggregating across a funding stream needs to know which results to weight heavily and which to interpret with caution.

This is a gap most agencies aren't addressing — and it's arguably the most important question for funders: how much should I trust these outcomes?

CIDS addresses this through the W3C Data Quality Vocabulary (DQV), which provides properties on IndicatorReport for signalling the quality of the data behind the numbers.

Our approach follows a principle from evaluation methodology: describe, don't rank. A published instrument isn't inherently better than a custom measure designed with the community it serves. A staff-observed rating isn't inherently more reliable than a self-report. What matters is that funders have enough context to make their own judgment for their specific use case. KoNote emits two types of DQV signals:

Quantitative measurements (computed automatically)

SignalWhat it tells fundersExample
Reporting rate What proportion of eligible participants actually have data? 19 of 23 eligible participants reported (83%)
Data parsability Were recorded values usable, or were many unparseable? 98% of values parsed successfully
Observation density How many data points per participant underpin the statistics? 14.9 observations per participant

Structured descriptors (declared by agency)

These describe how data is generated — they're set once per metric and don't imply a quality ranking:

DescriptorWhat it describesOptions
Evidence type How is the data generated? Self-report · Staff-observed · Administrative record · Third-party assessed · Coded qualitative
Measure basis How was the measure developed? Published validated · Published adapted · Custom participatory · Custom staff-designed · Administrative
Derivation method How was the value produced? Direct response · Coded from qualitative · Calculated composite · Staff rating

These signals use standard DQV properties (dqv:hasQualityMeasurement and dqv:hasQualityAnnotation on cids:IndicatorReport), so any system consuming CIDS exports can read them without custom parsing.

Why this matters for aggregation: When a funder rolls up outcomes across 15 funded programs, knowing that Program A has an 83% reporting rate using a staff-observed measure adapted from a published instrument, and Program B has a 25% reporting rate using a custom self-report scale, fundamentally changes how you interpret and weight the results. The signals don't say which is "better" — they give the funder enough context to decide for themselves.

How funders aggregate across agencies

A natural question: if each agency defines their own metrics, how does a funder produce a summary across an entire funding stream? The answer is that CIDS doesn't require agencies to use the same measures — it requires them to tag their measures with shared taxonomy codes. Aggregation happens at the taxonomy level, not the metric level.

How it would work in practice

  1. Funder defines a reporting template — expressed in CIDS vocabulary: "For this funding stream, report on SDG 8 outcomes, IRIS+ OI4060 indicators, stakeholder breakdowns by PopulationServed codes, and an impact model with risks and counterfactuals." The template specifies which taxonomy codes and CIDS classes the funder wants populated — not what specific metrics to use.
  2. Agency maps their existing metrics to the template — they already track employment outcomes in whatever way makes sense for their program. The mapping step is tagging their indicators with the codes the funder asked for (IRIS+ OI4060, SDG 8.5, etc.). KoNote's taxonomy classification workflow supports this.
  3. System generates a report against the template — the export filters to what the funder asked for, structured the way they asked for it.
  4. Funder receives structured data they can aggregate — because every agency reported against the same template (same codes, same classes), the funder can roll up across their funding stream by taxonomy code.

This means the funder can always answer "how many of our funded programs achieved their own employment targets?" without forcing everyone to measure employment the same way. One agency tracks job placement rates, another tracks hours of paid work, another tracks income change — but they're all tagged SDG 8 and IRIS+ OI4060, so the funder can group and compare them thematically.

Open question for Common Approach: Does the standard envision funders publishing reporting templates as CIDS documents — a set of required Indicators, Themes, and Codes without values filled in? This would be the contract between funder and grantee: the funder defines what to report on (taxonomy codes, CIDS classes), and the agency returns it with their data. That "reporting template" concept is the missing bridge between individual agency exports and funder aggregation.

Our mapping and modelling experience

You asked for detail on our experience mapping the CIDS classes and properties. Here's what we found:

  • Organization, Outcome, Indicator, IndicatorReport — mapped directly to our existing data (agency profile, outcome targets, metric definitions, aggregated observations)
  • Theme and Code — we can import the ICNPO, SDG, IRIS+, and PopulationServed code lists and link metrics and programs to specific codes
  • Stakeholder — maps to population served codes on our programs, which already describe the groups served (e.g., "youth aged 16-24")
  • ImpactModel, Service, Activity, ImpactRisk, Counterfactual — these don't exist as standalone entities in a case management system, so we built an evaluation framework editor where agencies define structured components that map to these classes. An agency creating their evaluation framework is simultaneously populating their CIDS export.
  • StakeholderOutcome — derived from group-level outcome statistics (success rates, means, distributions, percentiles) rather than individual records
  • IndicatorReport values — we store individual observations in a flexible format (to handle scales, achievement status, and open text), so we compute aggregates per metric type: success rates for achievement metrics, means for scale metrics
