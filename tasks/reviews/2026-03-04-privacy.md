## 4. Critical Findings

### Finding C-1: Communication model lacks retention expiry field {#finding-c-1}

**Severity:** MEDIUM
**File:** `apps/communications/models.py`
**PIPEDA Principle:** 5 (Limiting Use, Disclosure, and Retention)

The Communication model has no `retention_expiry`, `auto_purge_after`, or similar field. While client erasure (ErasureRequest) handles deletion of all client data including communications, there is no mechanism for time-based retention limits on communications independent of full client erasure. Communications are retained indefinitely until the client record is erased.

**Recommendation:** Add a retention-aware field or implement a periodic cleanup management command that enforces agency-defined retention periods for communications.

---

### Finding C-2: Registration email hash uses plain SHA-256, not keyed HMAC {#finding-c-2}

**Severity:** MEDIUM
**File:** `apps/registration/models.py`
**PIPEDA Principle:** 7 (Safeguards)

The RegistrationSubmission model computes `email_hash` as:
```python
self.email_hash = hashlib.sha256(email.encode()).hexdigest()
```

This is a plain, unkeyed SHA-256 hash. By contrast, the portal's ParticipantUser model uses HMAC-SHA-256 with a secret key:
```python
hmac.new(key, email, hashlib.sha256).hexdigest()
```

Plain SHA-256 hashes of email addresses are vulnerable to rainbow table attacks since email addresses are low-entropy and enumerable. An attacker with database access could reverse the hashes by hashing known email addresses.

**Recommendation:** Migrate RegistrationSubmission to use HMAC-SHA-256 with `EMAIL_HASH_KEY`, consistent with the portal's approach. Requires a data migration to rehash existing values.

---

### Finding C-3: EMAIL_HASH_KEY has weak demo-mode fallback {#finding-c-3}

**Severity:** LOW (production impact depends on deployment)
**File:** `konote/settings/base.py`
**PIPEDA Principle:** 7 (Safeguards)

`EMAIL_HASH_KEY` falls back to `"demo-email-hash-key-not-for-production"` when not set in the environment. While the key name signals it is not for production, the application does not enforce this -- a production deployment with a missing environment variable would silently use the weak fallback.

**Recommendation:** Add a Django system check (similar to `konote.E001` for FIELD_ENCRYPTION_KEY) that fails at boot if `EMAIL_HASH_KEY` equals the demo fallback and `DEMO_MODE` is False.

---

### Finding C-4: Registration submissions lack explicit audit trail {#finding-c-4}

**Severity:** LOW
**File:** `apps/registration/views.py`
**PIPEDA Principle:** 1 (Accountability)

The public registration view creates RegistrationSubmission records but does not write an explicit audit log entry. By contrast, the public survey view explicitly logs survey submissions to the audit database with a comment about PIPEDA 4.4.

The general AuditMiddleware will capture the POST request itself, but without resource_type or resource_id specificity for registration submissions.

**Recommendation:** Add explicit audit logging for registration submissions, consistent with the survey approach. Include submission ID but not PII (consistent with PIPEDA 4.4 approach used in surveys).

---

### Finding C-5: AuditLog old_values/new_values may contain PII {#finding-c-5}

**Severity:** LOW (architectural awareness)
**File:** `apps/audit/models.py`
**PIPEDA Principle:** 4 (Limiting Collection) / 7 (Safeguards)

The AuditLog model has `old_values` and `new_values` JSONFields. If model signals write field-level change data to these fields, they may contain decrypted PII (e.g., "old name: Jane, new name: Janet"). The audit database has its own access controls, but this creates a second location where decrypted PII exists at rest.

The audit database is stored separately and the AuditLog is immutable, which limits the attack surface. However, the retention of PII in audit entries means the audit database itself becomes subject to PIPEDA retention requirements.

**Recommendation:** Document whether `old_values`/`new_values` contain PII and if so, (a) ensure the audit database is encrypted at rest, and (b) include audit database PII in the agency retention policy. Consider storing field change indicators ("first_name changed") rather than actual values where possible.

### Finding C-6: RegistrationSubmission.field_values stored as plain JSON {#finding-c-6}

**Severity:** MEDIUM
**File:** `apps/registration/models.py`
**PIPEDA Principle:** 7 (Safeguards)

Custom field values in `RegistrationSubmission.field_values` are stored as a plain JSONField, not encrypted. Depending on the agency's custom field definitions, these values may contain sensitive personal information (e.g., health conditions, Indigenous status, disability information).
