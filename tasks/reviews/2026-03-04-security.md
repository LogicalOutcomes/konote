## Executive Summary

KoNote demonstrates a **mature security posture** for a healthcare-adjacent web application storing encrypted PII. The defence-in-depth architecture -- Fernet encryption for PII fields, program-scoped RBAC middleware, separate audit database, PHIPA cross-program consent enforcement, and DV safety controls -- is well-designed and consistently applied across the codebase.

All 11 gate checks pass. No critical (Sev-1) findings were identified. Three medium (Sev-2) findings and five low (Sev-3) findings are documented below. The most actionable items are:

1. **[Sev-2] Dead variable in encryption self-test** -- `_master_fernet` instead of `_fernet` in the `finally` block leaves the Fernet cache in an indeterminate state after the system check.
2. **[Sev-2] CSP `unsafe-inline` for scripts** -- XSS mitigation is weakened by allowing inline scripts for Chart.js initialisation.
3. **[Sev-2] EMAIL_HASH_KEY empty string default** -- If not set and not in DEMO_MODE, HMAC operations use an empty key, potentially allowing email enumeration in portal lookups.

### Finding Details

**F4 -- Inconsistent token entropy:**
`SurveyLink.token` uses `secrets.token_urlsafe(32)` (~192 bits). `PortalInvite.token` and `StaffAssistedLoginToken.token` use `secrets.token_urlsafe(48)` (~288 bits). `CalendarFeedToken.token` uses `secrets.token_urlsafe(32)`. While 192 bits is cryptographically sufficient, the inconsistency suggests an ad-hoc approach. Standardising on 48 bytes would be more defensible.

**F5 -- CSP unsafe-inline for styles:**
Lower severity than F2 because style injection alone cannot execute arbitrary code in modern browsers. Documented as a Pico CSS requirement with a note to remove it when Pico is vendored locally.

**F6 -- Public survey rate limit:**
30 requests per hour per IP may allow a modest spam attack against high-traffic survey links. The honeypot field and single-response cookie provide additional protection, but IP-based rate limiting is the primary defence. Consider a higher threshold or CAPTCHA for public-facing endpoints.

**F7 -- X-Forwarded-For trust:**
`get_client_ip()` (`konote/utils.py:4-8`) takes the first comma-separated value from `HTTP_X_FORWARDED_FOR`. In a multi-proxy environment, the first value is client-supplied and can be spoofed. Since this is used for audit logging (not access control), impact is limited to audit log accuracy. A more robust approach would strip to the last trusted proxy hop. However, the standard Railway/Azure proxy architecture typically provides a single reliable forwarded address.

**F8 -- Calendar feed token lifetime:**
`CalendarFeedToken` has no `expires_at` field. Tokens remain active until manually revoked via the settings page or regenerated. If a user's token is leaked (e.g., through a shared calendar app), it provides ongoing read access to their meeting schedule (initials + record IDs only, no full names). Adding an optional expiry or periodic rotation reminder would reduce this risk.

## Positive Security Patterns (Notable Strengths)

These design decisions represent security engineering above baseline expectations:

1. **Two-person rule for DV flag removal** (`dv_views.py`) -- Prevents a single compromised or coerced account from exposing a DV-safe client.

2. **Fail-closed PHIPA consent filtering** (`programs/access.py`) -- Cross-program notes are hidden by default. Consent must be explicitly recorded before clinical data crosses program boundaries.

3. **Separate audit database with INSERT-only permissions** (`db_router.py`, `lockdown_audit_db.py`) -- Even a full application database compromise cannot tamper with audit logs.

4. **Portal timing equalisation** (`portal/backends.py`) -- `make_password("dummy-timing-equalisation")` on email miss prevents username enumeration via timing side channels.

5. **DV small-circle hiding** (`circles/helpers.py`) -- When ClientAccessBlock hides a circle member and fewer than 2 enrolled participants remain visible, the entire circle is hidden to prevent identification by inference.

6. **PII scrubbing before AI API calls** (`ai_views.py`, `pii_scrub.py`) -- Two-pass scrubbing (regex + known names, longest-first) before any data leaves the application.

7. **Emergency logout with sendBeacon** (`portal/views.py:271-297`) -- DV-safe panic button that works even if the user is navigating away. Session-bound single-use token compensates for `csrf_exempt`.

8. **Suppression with secondary suppression** (`suppression.py`) -- Not just cell-level suppression but automatic complementary suppression to prevent derivation by subtraction.

9. **Signed unsubscribe URLs with expiry** (`communications/services.py:208-218`) -- Django signing with `max_age=60 days` for tamper-proof, expiring unsubscribe links. No database lookup required for validation.

10. **Agency data export with AES-256-GCM** (`export_agency_data.py`) -- Passphrase-based encryption with 600K PBKDF2 iterations, Diceware passphrase generation, explicit instructions to communicate passphrase out-of-band.
